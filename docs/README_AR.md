## 🎯 هدف المشروع
• **ما هي المهمة التي يحلها التطبيق؟**
يسمح للمستخدمين بالتقاط صور للأماكن وحفظها مع تعليقات نصية مباشرة في قاعدة بيانات محلية (SQLite) للتخزين والعرض دون اتصال بالإنترنت.
• **لماذا هذه المهمة مهمة؟**
يوفر طريقة بسيطة لإنشاء مذكرات بصرية شخصية أو كتالوج للمواقع دون الحاجة إلى تخزين سحابي أو اتصال بالإنترنت.
• **الجمهور المستهدف:**
المسافرون، الباحثون، أو أي شخص يريد تسجيل المواقع بسرعة مع الصور والملاحظات.

## 📌 متطلبات التطبيق
### المتطلبات الوظيفية
• القدرة على تشغيل الكاميرا.
• القدرة على حفظ الصورة وتعليق نصي في قاعدة البيانات.
• القدرة على عرض قائمة بجميع السجلات المحفوظة.
• القدرة على عرض معلومات مفصلة (صورة + نص) في مربع حوار (Dialog).
### المتطلبات غير الوظيفية
• **الأداء:** وقت استجابة سريع لعمليات قاعدة البيانات وتحميل الصور.
• **سهولة الاستخدام:** واجهة بسيطة وبديهية مع زرين رئيسيين.
• **الموثوقية:** معالجة قوية لنتائج الكاميرا وحفظ البيانات بشكل مستقر.

## 🧠 الهندسة المعمارية العامة
• **النهج المختار:**
MVC (Model-View-Controller) - النهج الكلاسيكي لمشاريع Java البسيطة في أندرويد.
• **لماذا تم اختياره؟**
نظراً لبساطة المشروع، يوفر MVC هيكلاً واضحاً حيث تدير الـ Activities منطق الواجهة والتحكم، بينما تمثل فئات `HelperDB` و `Info` طبقة البيانات.
• **مكونات النظام الأساسية:**
- `MainActivity`: نقطة الدخول والتنقل.
- `AddToDB`: المتحكم لالتقاط الصور وحفظ البيانات.
- `ViewFromDB`: المتحكم لاسترجاع وعرض البيانات.
- `HelperDB`: إدارة قاعدة بيانات SQLite.
- `Info`: فئة نموذج البيانات.

## 🧩 وصف تفصيلي للفئات
### 📌 الفئة: MainActivity
**الدور:** الشاشة الرئيسية.
**المسؤولية:** توفر التنقل إلى وحدات "الإضافة" و "العرض".
**الطرق الرئيسية:**
- `onCreate()`: تهيئة الأزرار وإعداد مستمعي النقرات للتنقل.
**التفاعل مع الفئات الأخرى:** تشغيل `AddToDB` و `ViewFromDB` عبر Intents.

### 📌 الفئة: AddToDB
**الدور:** شاشة إدخال البيانات.
**المسؤولية:** تدير التفاعل مع الكاميرا، إدخال المستخدم (EditText)، والحفظ في قاعدة البيانات.
**الطرق الرئيسية:**
- `callCameraApp()`: تحضير الملف وتشغيل الكاميرا.
- `onActivityResult()`: استلام الصورة وعرضها في ImageView.
- `bSave.onClick()`: ضغط الصورة إلى مصفوفة بايتات وحفظها في SQLite.

### 📌 الفئة: ViewFromDB
**الدور:** شاشة عرض البيانات.
**المسؤولية:** جلب السجلات من SQLite وعرضها في ListView.
**الطرق الرئيسية:**
- `buildList()`: الاستعلام من قاعدة البيانات وتعبئة محول القائمة (Adapter).
- `onItemClick()`: عرض صورة كاملة وتعليق في AlertDialog.

### 📌 الفئة: Repository / Data Layer (HelperDB)
**الدور:** طبقة البيانات.
**المسؤولية:** إنشاء وإدارة مخطط قاعدة بيانات SQLite (جدول `Places`).

## 🔄 مخطط عمل التطبيق
1. يفتح المستخدم التطبيق (`MainActivity`).
2. ينقر المستخدم على "Add" -> تفتح شاشة `AddToDB`.
3. ينقر المستخدم على `ImageView` -> تفتح الكاميرا.
4. يلتقط المستخدم صورة -> تظهر الصورة في `ImageView`.
5. يدخل المستخدم تعليقاً وينقر على "Save" -> يتم تخزين البيانات في SQLite.
6. يعود المستخدم وينقر على "View" -> تعرض `ViewFromDB` القائمة.
7. ينقر المستخدم على عنصر في القائمة -> يعرض `AlertDialog` الصورة والنص.

## 🎨 تحليل UI/UX
• **لماذا تم تصميم الواجهة بهذه الطريقة؟** لتقليل التعقيد. أزرار كبيرة في الشاشة الرئيسية تؤدي إلى مهام واضحة.
• **المبادئ المستخدمة:**
- **البساطة:** لا توجد قوائم غير ضرورية.
- **التدفق المنطقي:** عملية خطوة بخطوة لإضافة البيانات.
- **إمكانية الوصول:** عناصر تفاعلية كبيرة.
• **ما الذي يمكن تحسينه؟** إضافة عرض معرض صور بدلاً من قائمة نصية بسيطة.

## ⚙️ العمل مع الخيوط (Threads)
• **التقنيات المستخدمة:**
- خيط واجهة المستخدم الرئيسي (لكل العمليات).
• **لماذا هذه الطريقة؟** بالنسبة لقاعدة بيانات محلية صغيرة، هذا مقبول، ولكن للبيانات الكبيرة يفضل استخدام `Executors` أو `Coroutines`.
• **كيفية منع المشاكل:**
- التجميد (ANR): يتم تجنبه لأن عمليات قاعدة البيانات سريعة جداً.
- تسرب الذاكرة: استخدام دورات حياة Activity القياسية.

## 💾 إدارة البيانات
• **موقع التخزين:** قاعدة بيانات SQLite محلية (`places.db`).
• **لماذا هذه الطريقة؟** الموثوقية، الهيكلة، والعمل دون إنترنت.
